# ODB Fork Migration Guide

This guide documents the process of migrating from a single repository with client-specific branches to a fork model where each client has their own repository.

# Fork Model Overview

# Structure

  • Upstream: rixxo/odb - The canonical source of truth
  • Client Forks: clientname/odb - One per client that needs variation

# What Lives Where

# Upstream (rixxo/odb)

  • ODB core code
  • Generic fixes and improvements that could benefit more than one client
  • Releases/tags

# Client Fork (clientname/odb)

  • Client-only behaviour, branding, hard client constraints
  • Anything contractually restricted or specific to their environment

# Migration Steps

# 1. Identify Client-Specific Branches

First, identify which branches are client-specific vs. generic feature branches:

Client-specific branches (need forks):

  • feature/Chisholm-Hunter-Version → chisholm-hunter/odb
  • feature/ODB-CH → chisholm-hunter/odb
  • feature/Kingfisher → kingfisher/odb
  • feature/ODB-MonsterPlumb → monsterplumb/odb

Generic branches (stay in upstream):

  • feature/RODB-* - Ticket-based features
  • feature/SENG-* - Ticket-based features
  • hotfix/* - Urgent fixes
  • release/* - Release preparation

# 2. Run the Migration Script 

cd odb
.\migrate-to-forks.ps1

For a dry run (no changes):

.\migrate-to-forks.ps1 -DryRun

# 3. Create Fork Repositories on Bitbucket

For each client fork:

  1. Create the repository on Bitbucket:

    • Go to Bitbucket workspace
    • Create new repository: clientname-odb
    • Set it as private (if needed)

  2. Push the fork:

    cd ../clientname-odb
git remote add origin https://bitbucket.org/rxmodules-development/clientname-odb.git
git push -u origin main

  3. Set up upstream remote:

    git remote add upstream https://bitbucket.org/rixxo/odb.git
git fetch upstream

# 4. Configure Branch Protection

For each repository (upstream and forks):

# Upstream (rixxo/odb)

  • Protect main and release/* from direct pushes
  • Require PRs with approval
  • Branch restrictions: only maintainers can create release/*, hotfix/*
  • Set up CODEOWNERS for core areas

# Client Forks (clientname/odb)

  • Protect main from direct pushes
  • Require PRs with approval
  • Allow maintainers to merge from upstream

# 5. Update Documentation

Update any documentation that references branches to reference fork repositories instead.

# Branching Rules

# In Upstream (rixxo/odb)

  • main (or master) = released, stable
  • develop (optional) = integration branch
  • feature/<ticket>-<short-desc> = short-lived work
  • release/<x.y.z> = release preparation
  • hotfix/<ticket>-<short-desc> = urgent fixes

# In Client Forks (clientname/odb)

  • main = tracks their live version
  • Avoid using feature/ for long-lived variants (keep feature/ short-lived only)

# Change Flow Rules

# Reusable Fix (Generic Improvement)

  1. Implement in upstream first (branch + PR)
  2. Tag and release upstream

  3. Pull/rebase/merge into client forks as needed:

    cd clientname-odb
git fetch upstream
git checkout main
git merge upstream/main
# or
git rebase upstream/main

# Client-Only Change

  1. Implement in the client fork
  2. If any part becomes generic, split it into a separate upstream PR to main repo

# Syncing Forks with Upstream

Regular sync process for client forks:

# In client fork repository
git fetch upstream
git checkout main
git merge upstream/main
# Resolve any conflicts
git push origin main

Or using rebase (cleaner history):

git fetch upstream
git checkout main
git rebase upstream/main
# Resolve any conflicts
git push origin main --force-with-lease

# Troubleshooting

# Multiple Branches for Same Client

If a client has multiple branches (e.g., ODB-CH and Chisholm-Hunter-Version):

  1. Identify which is the primary/main branch
  2. Merge other branches into the primary before creating fork
  3. Or create fork from primary and manually merge other branches

# Conflicts When Syncing

When merging upstream changes into a fork:

  1. Resolve conflicts in favor of upstream (generic code)
  2. Re-apply client-specific changes after merge
  3. Document any persistent conflicts

# Missing History

If you need to preserve full history:

  • Use git subtree or git filter-branch to extract client-specific commits
  • Or keep the full history and document which commits are client-specific

# Post-Migration Checklist

  • All client branches identified and mapped
  • Fork repositories created on Bitbucket
  • Fork repositories pushed to Bitbucket
  • Upstream remotes configured in forks
  • Branch protection rules configured
  • Team members have access to appropriate repositories
  • Documentation updated
  • CI/CD pipelines updated (if applicable)
  • Old client branches archived or deleted (after verification)

# Example: Creating a New Client Fork

If you need to create a new client fork in the future:

  1. Create the fork repository on Bitbucket

  2. Clone upstream and create fork:

    git clone https://bitbucket.org/rixxo/odb.git clientname-odb
cd clientname-odb
git remote rename origin upstream
git remote add origin https://bitbucket.org/rxmodules-development/clientname-odb.git
git push -u origin main
  3. Make client-specific changes in the fork
  4. Document client-specific requirements

# Questions?

If you encounter issues during migration:

  1. Check this guide first
  2. Review the migration script output
  3. Consult with the team lead
  4. Document any new patterns or issues for future reference
