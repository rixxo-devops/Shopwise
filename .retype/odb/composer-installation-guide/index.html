# Composer Installation with Fork Repositories

# Current Setup

You're currently installing ODB via Composer, likely something like:

{
    "require": {
        "rixxo/odb": "^2.0"
    }
}

Or via a private repository:

{
    "repositories": [
        {
            "type": "vcs",
            "url": "https://github.com/rixxo-devops/ODB.git"
        }
    ],
    "require": {
        "rixxo/odb": "dev-master"
    }
}

# New Setup: Client-Specific Repositories

With the fork model, you have two options:

# Option 1: Install Client-Specific Repository (Recommended)

For client projects that need customizations, install the client-specific repository instead of the main ODB:

For Kingfisher projects:

{
    "repositories": [
        {
            "type": "vcs",
            "url": "https://github.com/rixxo-devops/kingfisher-ODB.git"
        }
    ],
    "require": {
        "rixxo/odb": "dev-main"
    }
}

For MonsterPlumb projects:

{
    "repositories": [
        {
            "type": "vcs",
            "url": "https://github.com/rixxo-devops/monsterplumb-odb.git"
        }
    ],
    "require": {
        "rixxo/odb": "dev-main"
    }
}

Benefits:

  • ✅ Gets client-specific code automatically
  • ✅ No need to override or patch
  • ✅ Clean separation
  • ✅ Easy to update client-specific version

# Option 2: Install Main ODB (For Generic Projects)

For projects that don't need client customizations, continue installing the main ODB:

{
    "repositories": [
        {
            "type": "vcs",
            "url": "https://github.com/rixxo-devops/ODB.git"
        }
    ],
    "require": {
        "rixxo/odb": "dev-master"
    }
}

# Authentication

# Using Personal Access Token

If the repositories are private, you'll need authentication. Add to composer.json:

{
    "repositories": [
        {
            "type": "vcs",
            "url": "https://YOUR_TOKEN@github.com/rixxo-devops/kingfisher-ODB.git"
        }
    ]
}

Better approach: Use auth.json (don't commit this):

{
    "http-basic": {
        "github.com": {
            "username": "YOUR_TOKEN",
            "password": "x-oauth-basic"
        }
    }
}

Or use SSH (if you have SSH keys set up):

{
    "repositories": [
        {
            "type": "vcs",
            "url": "git@github.com:rixxo-devops/kingfisher-ODB.git"
        }
    ]
}

# Workflow Example

# Setting Up a New Kingfisher Project

  1. Create composer.json:
{
    "repositories": [
        {
            "type": "vcs",
            "url": "https://github.com/rixxo-devops/kingfisher-ODB.git"
        }
    ],
    "require": {
        "rixxo/odb": "dev-main"
    }
}
  1. Install:
composer install
# or
composer require rixxo/odb:dev-main
  1. Update when needed:
composer update rixxo/odb

# Syncing Client Fork with Upstream

When you want to pull updates from main ODB into the client fork:

cd /path/to/kingfisher-ODB
git fetch upstream
git merge upstream/master
# Resolve conflicts, test
git push origin main

Then in client projects:

composer update rixxo/odb

# Branch/Tag Strategy

# Using Branches (Current) 

{
    "require": {
        "rixxo/odb": "dev-main"  // Uses main branch
    }
}

# Using Tags (Recommended for Production)

Tag releases in each repository:

# In kingfisher-ODB repo
git tag -a v2.1.0-kingfisher -m "Kingfisher version 2.1.0"
git push origin v2.1.0-kingfisher

Then in composer.json:

{
    "require": {
        "rixxo/odb": "^2.1.0-kingfisher"
    }
}

# Comparison: Before vs After

# Before (Branches) 

{
    "repositories": [
        {
            "type": "vcs",
            "url": "https://github.com/rixxo-devops/ODB.git"
        }
    ],
    "require": {
        "rixxo/odb": "dev-feature/Kingfisher"  // Branch name
    }
}

Issues:

  • ❌ Branch names can change
  • ❌ Hard to track which branch is for which client
  • ❌ All branches in one repo (clutter)

# After (Fork Repositories) 

{
    "repositories": [
        {
            "type": "vcs",
            "url": "https://github.com/rixxo-devops/kingfisher-ODB.git"
        }
    ],
    "require": {
        "rixxo/odb": "dev-main"  // Always main branch
    }
}

Benefits:

  • ✅ Clear which client repo to use
  • ✅ Always use main branch (simpler)
  • ✅ Can use tags for versioning
  • ✅ Cleaner, more maintainable

# Multiple Client Repos in Same Project

If you need to support multiple clients in one project (unlikely, but possible):

{
    "repositories": [
        {
            "type": "vcs",
            "url": "https://github.com/rixxo-devops/ODB.git"
        },
        {
            "type": "vcs",
            "url": "https://github.com/rixxo-devops/kingfisher-ODB.git"
        },
        {
            "type": "vcs",
            "url": "https://github.com/rixxo-devops/monsterplumb-odb.git"
        }
    ],
    "require": {
        "rixxo/odb": "dev-main"
    }
}

Composer will use the first matching repository.

# Summary

Yes, Composer installation still works perfectly!

  • ✅ Can install from any of the repositories
  • ✅ Same Composer syntax
  • ✅ Just point to the client-specific repo URL
  • ✅ Can use branches or tags
  • ✅ Authentication works the same way

The only change: Point Composer to the client-specific repository instead of the main ODB repository for client projects.
