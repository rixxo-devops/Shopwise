# ODB Fork Model: Explanation and Implementation

# The Goal

Move from client-specific branches to separate fork repositories for client customizations of ODB.

Current State:

  • ODB installed via Composer
  • Client customizations done as branches in the main ODB repo
  • All branches in one repository (rxmodules-development/odb)

Desired State:

  • Each client has their own repository
  • Client-specific code isolated from main ODB
  • Easier to manage client-specific changes
  • Clear separation of concerns

# The Challenge: GitHub Fork Limitations

# GitHub's Native Fork Feature

GitHub's "Fork" button creates a UI relationship between repositories:

  • Shows "forked from..." in the repository UI
  • Enables easy pull requests back to the original
  • Limitation: Cannot fork within the same organization

Why this matters:

  • We typically work within rixxo-devops organization
  • We don't put repositories in client organizations
  • Therefore, GitHub's native fork feature cannot be used

# Our Solution: Manual Fork Model

# What We've Implemented

We've created separate repositories with manual upstream configuration:

rixxo-devops/
  ├── ODB (upstream, source of truth)
  ├── kingfisher-ODB (client fork)
  └── monsterplumb-odb (client fork)

Each client repository has:

  • origin remote → points to itself
  • upstream remote → points to rixxo-devops/ODB

# How It Works

Syncing from upstream (same as forking):

cd kingfisher-ODB
git fetch upstream
git merge upstream/main
# or
git rebase upstream/main

Making client-specific changes:

# Work directly in the client repo
git checkout -b feature/client-specific-change
# Make changes, commit, push

Pulling generic improvements from upstream:

git fetch upstream
git merge upstream/main
# Resolve conflicts, push

# Why This Achieves the Same Thing

GitHub Native Fork Our Manual Fork Model
Shows "forked from" in UI No UI indicator (but functionally identical)
Can create PRs to upstream Can create PRs to upstream (same org)
Automatic upstream tracking Manual upstream remote (same result)
Can't fork same org Works within same org ✅
Requires separate org Works in our org ✅

Key Point: The workflow is identical. The only difference is the GitHub UI doesn't show the fork relationship, but all the functionality works the same.

# If We Want True GitHub Forks

# The Requirement

To use GitHub's native fork feature, we would need to:

  1. Host forks in client organizations (not rixxo-devops)
  2. Each client would need their own GitHub organization
  3. We'd fork rixxo-devops/ODB → client-org/ODB

Example:

rixxo-devops/ODB (upstream)
    ↓ (GitHub fork)
kingfisher-org/ODB (client fork)
monsterplumb-org/ODB (client fork)

# Why This Typically Doesn't Work for Us

  1. Client organizations don't exist:

    • Clients don't have GitHub organizations
    • We don't want to create/manage client GitHub accounts
    • Clients don't need direct GitHub access

  2. Composer installation:

    • ODB is installed via Composer in client projects
    • Composer pulls from repositories
    • Having repos in client orgs would require:
      • Managing client GitHub credentials
      • Setting up authentication for each client
      • More complex deployment process

  3. Our workflow:

    • We maintain all code in rixxo-devops
    • We control access and permissions
    • We don't want to distribute repos across multiple organizations

# Benefits of Our Approach

# ✅ Works Within Same Organization

  • All repos in rixxo-devops
  • No need for client organizations
  • Centralized management

# ✅ Same Functionality as Forks

  • Can sync from upstream
  • Can merge upstream changes
  • Can create PRs (within same org)
  • Isolated client code

# ✅ Better Than Branches

  • Clear separation per client
  • No branch clutter in main repo
  • Easier to manage client-specific code
  • Can set different permissions per client repo

# ✅ Composer Compatible

  • Can still install via Composer
  • Points to rixxo-devops/ODB or client-specific repo
  • No authentication complexity

# Conclusion

Our manual fork model achieves the same goals as GitHub forking:

  • ✅ Isolated client repositories
  • ✅ Ability to sync from upstream
  • ✅ Clear separation of concerns
  • ✅ Works within our organization
  • ✅ Compatible with our workflow

The only difference: GitHub's UI doesn't show a "forked from" indicator, but functionally it works identically.

If we want true GitHub forks: We'd need to host repositories in client organizations, which doesn't fit our typical workflow and Composer-based installation process.

# Recommendation

Stick with our manual fork model because:

  1. It achieves all the same goals
  2. Works within our organization
  3. Compatible with Composer
  4. Easier to manage centrally
  5. No need for client GitHub organizations

The lack of a GitHub UI fork indicator is a minor cosmetic difference that doesn't affect functionality.
